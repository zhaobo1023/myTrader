## 一、系统全景

### 1.1 目标

将专业分析师（以 f大/Flitter 为基准）的财报分析方法论编码到系统中，让 AI 研报从"看到好数字就说好"进化到"拆解每个数字的可信部分和操作部分"。

### 1.2 架构总览

```
年报 PDF
    |
    v
[MDConverter]  ---- pymupdf4llm / MinerU 3.0
    |
    v
Markdown (.md)  ---- 统一中间格式，一式两用
    |
    +--> Path A: Embedding --> ChromaDB --> RAG 检索 (定性内容：管理层讨论、风险披露)
    |
    +--> Path B: 结构化提取 --> MySQL (定量数据：利润表、逾期贷款、资本结构)
         |
         +--> 正则/表格解析 (BankTemplate) --> financial_income_detail / bank_overdue_detail
         |
         +--> LLM 关键指标提取 (KeyMetricsExtractor) --> 行业可配置指标 (OCI/永续债/拨备余额)
    |
    v
评分卡 / 诊断信号 / 研报生成
```

### 1.3 数据库表结构

| 表名 | 用途 | 数据来源 |
|------|------|---------|
| `financial_income` | 利润表摘要 (营收/净利/ROE) | AKShare |
| `financial_balance` | 资产负债表 + 银行指标 (NPL/NIM/CAR) | AKShare |
| `financial_dividend` | 分红历史 | AKShare |
| `financial_income_detail` | 利润表明细 (非息收入/OCI/信用减值) | 年报 PDF 提取 |
| `financial_cashflow` | 现金流摘要 | 年报 PDF 提取 |
| `bank_overdue_detail` | 逾期贷款明细 + NPL率2 | 年报 PDF 提取 + f大点评手工补录 |
| `trade_stock_daily` | 日线行情 (价格/PB/PE) | QMT/Tushare |

### Phase 3.5：评分卡偏差修复 (2026-04-13)

**目标**：基于 14 家银行评分卡 vs f大对比分析，修复 5 个系统性偏差。

**偏差分析**：
对 14 家银行运行评分卡 + LLM 关键指标提取，逐维度对比 f大评价，发现 5 个系统性偏差：
1. OCI 推算被永续债扭曲 (民生 1250 亿永续债导致 D4 虚高)

| D7 权重 | 10% -> 7.5% | 股息率已纳入 D6，避免重复 |
| D4 权重 | 20% -> 17.5% | 给 D2/D6 让出权重 |

- `_fetch_real_oci()` -- 从 financial_income_detail 查询 LLM 提取的真实 OCI

## 三、核心模块详解

### 3.1 年报提取器 (`annual_report_extractor.py`)

**流程**：
```
PDF --> [MDConverter] --> .md
    --> [BankTemplate] --> income_detail + overdue_detail + cashflow_detail
    --> [KeyMetricsExtractor] --> key_metrics (OCI/永续债/拨备余额/信用减值)
    --> [FinancialStorage.upsert()] --> MySQL (自动合并去重)
```

**使用方式**：
```python
from data_analyst.financial_fetcher.annual_report_extractor import AnnualReportExtractor

extractor = AnnualReportExtractor(db_env='online')

# 单文件处理
extractor.process_md("华夏银行_600015_2025_年报.md", industry="bank")

# PDF 自动转换
extractor.process_pdf("华夏银行_600015_2025_年报.pdf", industry="bank")

# 批量处理
extractor.batch_process_dir("annual_reports/600015/", industry="bank")
```

**输出表**：

| 表 | 提取方式 | 关键字段 |
|----|---------|---------|
| `financial_income_detail` | 正则 + LLM 合并 | 非息收入/OCI/永续债/拨备余额/信用减值 |
| `bank_overdue_detail` | 正则 | 逾期90+/重组贷款/NPL率2/总贷款 |
| `financial_cashflow` | 正则 | 经营/投资/筹资现金流 |

### 3.2 银行评分卡 (`bank_scorecard.py`)

**7 维度打分体系 (v2.2)**：

| 维度 | 名称 | 权重 | 子维度 | 数据来源 |
|------|------|------|--------|---------|
| D1 | 资产质量 | 25% | NPL率(0.15) / 拨备覆盖率(0.20) / NPL2-官方差值(0.25) / NPL2变化趋势(0.40) | bank_overdue_detail |
| D2 | 盈利能力 | 15% | ROE(0.50) / NIM(0.30) / 归母净利同比增速(0.20) | financial_income + financial_balance |
| D3 | 利润质量 | 15% | 拨备释放贡献(0.30) / 拨备比变化趋势(0.40) / 剪刀差(0.30) | 推算 |
| D4 | 净资产质量 | 17.5% | OCI差值占比(0.40) / 净资产增速(0.60) | financial_income_detail (真实) + 推算 (回退) |
| D5 | 资本充足 | 10% | 一级资本充足率(0.40) / 资本压力(0.60) | financial_balance |
| D6 | 估值安全边际 | 15% | PB历史分位(0.30) / 同业PB相对位置(0.30) / 股息率vs国债(0.20) / PBx股息率(0.20) | trade_stock_daily_basic + financial_dividend |
| D7 | 股息回报 | 7.5% | 股息率(0.60) / 分红增长(0.40) | financial_dividend + 行情 |

**评分卡 vs f大评价**：
- 评分卡用"绝对水平 + 趋势"混合框架
- f大用纯"边际惊喜度"框架 (实际 vs 推算值的偏离方向和幅度)
- OLS 逆向工程 R^2=0.794，权益惊喜权重最大 (+0.925)
- 高分端仍有偏差，需要进一步向"纯边际惊喜"模式切换

### 3.3 LLM 关键指标提取器 (`key_metrics_extractor.py`)

**行业配置** (`key_metrics_config.yaml`)：

**银行 (7 个指标)**：
- 其他综合收益 (OCI) -- FVOCI 债券浮盈浮亏
- 综合收益总额 -- 净利润 + OCI
- 其他权益工具余额 -- 永续债 + 优先股
- 其他权益工具应付利息 -- 永续债应付未付利息
- 贷款减值准备余额 -- 拨备覆盖率的分母
- 净利息收入
- 信用减值损失

**消费品 (5 个指标)**：毛利率 / 销售费用率 / 经销商数量 / 存货余额 / 合同负债

**制造 (4 个指标)**：研发费用 / 研发费用率 / 产能利用率 / 在建工程

**扩展新行业只需在 YAML 中添加配置**，代码无需修改。

## 四、对比分析：Qwen vs Claude vs f大

### 4.1 测试场景

以华夏银行 (600015) 2025 年报为分析对象，对比三种分析方式的产出差异。

### 4.2 f大分析方法论核心

f大 (雪球@flitter) 的分析方法可以总结为**数字拆解法**：

```
报告净利润 = 真实经营利润 + 拨备释放 + 税收节约 + OCI影响
报告净资产 = 真实资产价值 + OCI浮盈浮亏
报告NPL率  = 真实不良 x 认定严格系数
```

核心思维：**看到 272 亿净利润，不看数字大小，而是拆解每个组成部分有没有变好**。

**六大强制问题**：
1. 不良是真实的吗？(NPL率2 vs 官方NPL率)
2. 利润是真实的还是靠拨备撑的？(拨备释放贡献占比)
3. 净资产是否被隐性侵蚀？(OCI差值)
4. 资本够用吗？(贷款增速 vs 内生资本补充)
5. 息差是在稳住还是在塌陷？(NIM趋势)
6. 市场定价了什么？(PB隐含ROE vs 实际ROE)

### 4.4 通用 LLM 研报 (Qwen/Claude) 的典型问题

**问题 1：只看表面数字**
- LLM 看到"净利润 272 亿，同比增长 X%"，直接得出"盈利能力强劲"
- f大 看到 272 亿，先问"其中多少是拨备释放？多少是 OCI？"

**问题 2：缺乏横向对比意识**
- LLM 单独分析一家银行，不与同业对标
- f大 分析华夏银行时，必定与 17 家银行做横截面对比

**问题 3：估值分析表面化**
- LLM："PB 0.4x 处于历史低位，估值有吸引力"
- f大："PB 0.4x 对应隐含 ROE 只有 6%，但实际 ROE 10%+，结合股息率 7%+，即使利润零增长也有超额回报"

**问题 4：不会拆解利润来源**
- LLM 把净利润当作一个整体指标
- f大 的利润公式：`真实利润 = 公告利润 + NPL率2变动 x 贷款 x 0.75 + 拨备比变动 x 贷款 x 0.75`

**问题 5：对 OCI 和永续债不敏感**
- 通用 LLM 通常忽略"其他综合收益"和"其他权益工具"这两个科目
- f大 把它们作为核心扣除项：`真实净资产 = 股东权益 - 全部其他权益工具 - 永续债应付利息`

### 4.5 本次迭代后的改进效果

| 维度 | 通用 LLM 研报 | 加行业配置后 | 加评分卡 v2.1 | 加年报提取后 | v2.2 偏差修复 |
|------|-------------|------------|--------------|------------|--------------|
| 资产质量 | 看官方 NPL 率 | 提到 NPL率2 概念 | 量化打分 + 趋势 | 从年报提取真实值 | 不变 |
| 利润质量 | 整体评价 | 拨备释放概念 | 量化拨备贡献占比 | 获取拨备余额精确计算 | 不变 |
| 净资产质量 | 忽略 OCI | OCI 差值概念 | 量化 OCI 占比 (推算) | 从年报提取真实 OCI | 优先用真实 OCI |
| 盈利能力 | 整体评价 | ROE/NIM 概念 | 量化 (权重10%) | 不变 | 权重提升到 15% |
| 估值分析 | PB 百分位 | PB-ROE 框架 | PB 百分位 (权重10%) | 精确 PB 数据 | 4 子维综合 (权重15%) |
| 横向对比 | 无 | 同业对标文字 | 17 家银行排名表 | 数据完整支撑排名 | 同业 PB 对比纳入 D6 |

## 五、经验教训

### 5.1 数据层面

**教训 1：stock_code 格式不一致是最大的坑**

不同数据库表使用不同格式：
- `trade_stock_daily` / `trade_stock_daily_basic`：`600015.SH`
- `financial_balance` / `financial_income` / `bank_overdue_detail`：`600015`

**后果**：JOIN 查询全部失败，WHERE 条件匹配不到数据。

**解决**：查询价格时用 `SUBSTRING_INDEX(stock_code, '.', 1) = %s`，查财务表用纯数字。

**教训 2：cash_div 单位不统一**

数据库存储的是每 10 股分红 (如 3.20)，不是每股分红 (0.32)。

**后果**：股息率计算偏差 10 倍。

**解决**：读取后统一 `/ 10` 转换为每股。

**教训 3：分红年度归属逻辑**

ex_date 在 1-7 月的分红属于上一年度年报的分红，不是当年。

**后果**：把中期分红算成上年分红，年度分红增长率算成 -75%。

**解决**：`month <= 7 -> fiscal_year = ex_year - 1`，并用 75% 阈值检测不完整财年。

**教训 4：易混淆银行代码**

601128 = 常熟银行 (NOT 渝农商行)，601077 = 渝农商行。

**后果**：评分卡给错银行打分，排名全部错误。

**解决**：在 memory 中记录易混淆代码。

### 5.2 方法论层面

**教训 5：LLM 不能做计算，只能做解读**

让 LLM 从年报中"计算 NPL率2"，它会频繁出错 (数字看错、单位混淆、正负号搞反)。

**解决**：代码层做计算，LLM 只做数值提取和结论解读。三层分工：
- 正则/表格解析：结构化表格数据 (准确、快、零成本)
- LLM 提取：非结构化深度指标 (OCI/永续债，正则覆盖不到的)
- 代码计算：所有派生指标 (NPL率2/OCI差值/拨备贡献占比)

**教训 6：绝对水平 vs 边际变化是本质差异**

f大评价银行的核心不是"谁绝对指标最好"，而是"谁超预期/低于预期最多"。

评分卡 v2.2 用"绝对水平 + 趋势"混合，和 f大的纯"边际惊喜度"仍有差距。v2.2 通过调整权重和增加估值子维度改善了部分偏差，但民生银行仍被高估（OCI 推算值扭曲）。

**解决方向**：未来评分卡 v3.0 应切换到纯边际惊喜度模式。

**教训 6b：估值需要多维综合，不能只看单指标**

v2.1 的 D6 只看 PB 历史百分位一个指标，导致华夏银行（PB 低 + 股息率高 + 实际增速快）被严重低估评分。v2.2 扩展为 4 个子维度后华夏从 65.7 升到 70.2。

**教训 6c：权重分配影响巨大**

D2 盈利能力从 10% 提到 15%，招商银行 (ROE 13.44%) 就升了 3.5 分。权重 5% 的差异可以导致排名变动 2-3 位。

**教训 7：OCI 数据是银行分析的"盲区"**

AKShare 不提供 OCI 数据，导致评分卡 D4 维度长期用推算值。但 OCI 对银行净资产影响巨大 (华夏银行 OCI -55.56 亿，占净利 20%)。

**解决**：通过年报 PDF 提取 (本次已实现)，LLM 从年报文本中精准提取。

### 5.3 工程层面

**教训 8：行业扩展应该是配置化的，不是代码化的**

如果每个行业都写一套提取代码，维护成本会爆炸式增长。

**解决**：`key_metrics_config.yaml` -- 新增行业只需加 YAML 配置：
```yaml
new_industry:
  label: "新行业"
  metrics:
    - name: "关键指标名"
      key: "db_field_name"
      unit: "yi"
      table: "target_table"
      description: "帮助 LLM 定位的说明"
      aliases: ["别名1", "别名2"]
      required: true
```

**教训 9：正则提取优先，LLM 补充**

正则/表格解析对结构化表格数据准确率 95%+，且零成本、毫秒级。LLM 提取准确率 80-90%，且需要 API 调用成本。

**分工原则**：
- 表格中的数据 (利润表/资产负债表/逾期贷款表) -> 正则
- 散落在文本中的数据 (OCI/永续债/管理层讨论中的前瞻指标) -> LLM

**教训 10：数据完整性检查要在评分前做**

评分卡 17 家银行中，宁波银行缺 2 个字段 (prov_release_contrib_pct, nim)，导致评分不完整。

**解决**：评分前检查数据完整度，缺失字段标记为 N/A 并在评分时降低该子维度权重。

### 5.4 分析质量层面

**教训 11：专业分析的壁垒不是"更多数据"，而是"拆解思维"**

通用 LLM 拿到同样多的数据，产出质量远不如专业分析师。核心差距在于：
- 分析师会问"这个数字的每一分钱从哪来" (拆解思维)
- LLM 倾向于"数字大就是好" (表面思维)

**编码这个思维**的路径：
1. 路径 A (90% 效果)：强化 prompt，注入行业强制问题清单 (Phase 1 已完成)
2. 路径 B (更可靠)：计算层前置，预计算信号给 LLM 解读 (Phase 3 部分完成)
3. 路径 C (最完整)：评分卡体系，量化每个维度 (Phase 3 已完成 v2.1)
